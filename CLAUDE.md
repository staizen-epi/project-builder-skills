# CLAUDE.md — Spec-Driven Build Skills (project context)

Context for continuing this project in Claude Code. This repo is a **suite of Claude Code skills** that take a web app from idea → running foundation, spec-first. When extending it (new skills, edits, fixes), follow the conventions below so the set stays coherent.

`README.md` is the **user-facing** usage guide (how to talk to the skills). This file is the **builder/maintainer** context (how the skills are designed and how to add more). Keep that separation.

## Repo layout — the `package/` folder

The exportable payload lives under `package/`, so it can be copied wholesale into any new project. It currently holds:

- `package/skills/` — the ten skill folders (each a `SKILL.md`); this is the source of truth for the skills.
- `package/memory/` — seed memory files (with a `MEMORY.md` index) that travel with the package, so a project that adopts the skills starts with the right habits baked in (e.g. always check the project-local `.claude/skills/` first).

When exporting to another project, copy `package/skills/` into that project's `.claude/skills/` and `package/memory/` into its `.claude/memory/`. The repo root (`CLAUDE.md`, `README.md`) is maintainer context and does **not** get exported. Keep new exportable artifacts inside `package/`.

## What's here

Ten skills, each a folder with a `SKILL.md` (under `package/skills/`). They chain by reading/writing shared files:

| Skill | Stage | Reads | Writes |
|---|---|---|---|
| `product-requirements` | Define the product | conversation | `specs/PRD.md` |
| `feature-spec` | Detail one feature | `specs/PRD.md` | `specs/features/<feature>.md` |
| `quality-requirements` | Set non-functional targets | `specs/PRD.md` | `specs/QUALITY.md` |
| `web-app-architect` | Design the build | `specs/PRD.md` + `specs/QUALITY.md` (targets to design to) | `specs/ARCHITECTURE.md` |
| `poc-developer` | Spike a throwaway glimpse (greenfield) **or** quick-response spike of one change (brownfield) | `specs/ARCHITECTURE.md` (soft); brownfield also reads the **real code as context** | `/poc/` (throwaway) + `specs/POC-NOTES.md` |
| `ux-designer` | Design the look & feel (portable, project-agnostic) | `specs/PRD.md` (parent — persona/mood only) + `specs/POC-NOTES.md` + `specs/ARCHITECTURE.md` | `specs/DESIGN.md` + the portable bundle `specs/design/` (neutral tokens + adapters + buildless preview + `BUNDLE.md`) |
| `design-binder` | Bind the portable design system to *this* project | `specs/PRD.md` + `specs/ARCHITECTURE.md` + `specs/design/` (the bundle) | `specs/DESIGN-BINDING.md` (screen/flow→component map + chosen stack adapter) + a project theme-override token file under `specs/design/themes/` |
| `web-app-scaffolder` | Stand up Phase 0 | `specs/ARCHITECTURE.md` + `specs/POC-NOTES.md` (hints) + `specs/DESIGN.md` + `specs/DESIGN-BINDING.md` (adopted) | the codebase |
| `feature-developer` | Build one feature | `specs/features/<feature>.md` + `specs/ARCHITECTURE.md` (incl. §7a testID scheme) + `specs/REUSE.md` + `specs/DESIGN.md` + `specs/DESIGN-BINDING.md` | code + tests (testIDs per scheme) + `specs/REUSE.md` |
| `qa` | Verify & gate the build | `specs/ARCHITECTURE.md` (§7a testID scheme + test-env contract) + `specs/QUALITY.md` (NFRs) + `specs/PRD.md`/`specs/features/` (acceptance) + the running app | test code (in repo) + `specs/QA-PLAN.md` + `specs/QA-REPORT.md` |

Pipeline: **PRD → feature specs → (optional non-functional requirements) → architecture → (optional PoC) → (optional design system → design binding) → scaffold → build loop**. `quality-requirements` is the non-functional companion to `product-requirements`: it sits **after the PRD and alongside the architect**, owning the *how-well* (quality attributes + GDPR) as measurable `NFR-0X` targets in `specs/QUALITY.md` while the PRD owns the *what*. The seam with `web-app-architect` is **bidirectional**: `QUALITY.md` is the source of truth for the targets, the architect designs to them and records *how* in `ARCHITECTURE.md`, and an infeasible target routes back to `quality-requirements` to renegotiate (the target, not the mechanism, lives here). It's gated SaaS-aware — a SaaS gets a baseline quality floor (tenant-isolation, availability, perf, accessibility, observability, GDPR-on), a trivial/internal tool gates most dimensions off. Verification of each target is owned by the `qa` skill, which turns each `NFR-0X`'s `Verified by: qa` handoff into a runnable check. The build loop is the `feature-spec` → `feature-developer` pair, run once per feature: the writer details behaviour (and keeps the PRD §7 index pointing at it), the developer implements that behaviour as a vertical slice within the architecture and conforming to the design system. The PoC is an optional, disposable spike that runs in **two auto-detected modes**: **greenfield** (no real app yet) is the step between architecture and scaffold — it shows what the app could feel like on mock data and leaves `specs/POC-NOTES.md` as hints so the *scaffolder* starts closer to reality; **brownfield** (a real app already exists) is a **"quick response" spike** of one targeted change — it reads the real code as *context* to learn the integration seam, builds the change in `/poc` against a **mocked** version of that seam (never copying or touching real code), and hands off to **`feature-spec` → `feature-developer`** to build it for real. In both modes the `/poc/` code is throwaway — the next stage reads only `POC-NOTES.md`, never the spike code; in brownfield the notes additionally name the *real seam* (files/components/types) the change must wire into.

The **design layer is two skills with a deliberate seam — a portable artifact and a connecting artifact** (both optional and gated):
- **`ux-designer` → the portable design bundle.** It establishes the durable visual/interaction language as a **self-contained, project-agnostic bundle** (`specs/DESIGN.md` + `specs/design/`: stack-neutral tokens, optional adapters, a buildless static style-guide preview, and `BUNDLE.md`). It is deliberately **free of project specifics** (no flows, no screens, no brand instance) — that's what makes it **exportable**: copy the bundle into any project and it renders standalone with no build, app, or originating-repo dependency. It reads the PRD only for *persona and mood*, not to wire itself to the project.
- **`design-binder` → the connecting artifact.** It **hard-wires the PRD + ARCHITECTURE to the bundle**, contextualizing the generic system into *this specific project*: a screen/flow→component map (which bundle components/tokens/layout realize each PRD flow + architecture route), the project's **theme-override** token file (the brand instance, layered on the bundle without polluting it — lives under `specs/design/themes/`), and the **stack-adapter selection** (which bundle adapter the architecture's stack consumes). Output: `specs/DESIGN-BINDING.md`. This is the artifact the developers read to know *how this project uses* the inherited design system.

Together they give you **one exportable design artifact (inheritable into any project) and one connecting artifact (project-specific) that the developer skills consume**: `web-app-scaffolder` adopts the bundle (a decision) and the binding (the project theme + chosen adapter + shell); every `feature-developer` slice conforms to both (the bundle's components/tokens, mapped to its screen by the binding). Missing components/tokens route back to `ux-designer`; a wrong/missing screen→component mapping or theme override routes back to `design-binder`; gaps in either route like behaviour gaps route to `feature-spec`. All durable artifacts live under one `specs/` folder in the target project.

The **`qa` skill closes the loop** — it verifies the built app and owns the pass/fail gate before release. It tests the app as a **black box** (through the rendered UI + the published API; it never imports source or touches the DB), running four gated passes: **e2e feature/functionality** (always on), **smoke-set flagging** (always on; tags `@smoke` in-suite + lists the set), a **bounded, non-destructive API-vulnerability scan** (gated on the app exposing an API or holding sensitive/multi-tenant data; OWASP-style — authz/IDOR/cross-tenant, authn gaps, injection, headers, secret leakage, rate-limit presence — never DoS/destructive), and **performance testing** (gated on a perf `NFR-0X` existing in `QUALITY.md` to test against). It turns each `QUALITY.md` `NFR-0X` whose `Verified by:` names qa into a runnable check, requires the architecture's CI gate (lint/typecheck/unit) green **first**, then runs its passes and **restarts the full gauntlet on any failure until green**. The suite is deliberately **test-centered via a shared seam**: `web-app-architect` owns the **testID scheme** and the **test-environment contract** in `ARCHITECTURE.md` §7a (a build convention, like the ID strategy — depth scaled to the app, with an always-required reset hook for idempotent runs); `feature-developer` gives every interactive/meaningful element a **function-named** `data-testid` per that scheme (`download-button`, `quick-search-input`, `invoice-row-{id}` — named by *what it does*, with an `{id}` suffix for collection items); `qa` **derives** its selectors from the same scheme rather than scraping the DOM — so QA grabs each element by name instead of deep XPaths, every e2e selector is stable, and it still traces back to a requirement. qa owns running tests and the gate verdict only: it routes a behaviour bug to `feature-developer`, a missing/non-conforming testID to `feature-developer` (the testID) or `web-app-architect` (the scheme), a missing/infeasible target to `quality-requirements`, and a missing test-env/scheme to `web-app-architect` — it never patches feature code or weakens a check to force a green. It writes `specs/QA-PLAN.md` (the durable test contract — selector map, suite scope, smoke set, `NFR→check` map; the QA analogue of `REUSE.md`) and `specs/QA-REPORT.md` (the latest run's verdict).

```
specs/
├── PRD.md                  # what & why (highest level)
├── features/<feature>.md   # how each complex feature behaves (children of the PRD)
├── QUALITY.md              # non-functional targets (NFR-0X: perf, availability, security, a11y, GDPR…), maintained by quality-requirements (optional)
├── ARCHITECTURE.md         # how it's built (incl. §7a testID scheme + test-env contract — consumed by feature-developer & qa)
├── POC-NOTES.md            # hints from a throwaway spike — for the scaffolder (greenfield) or feature-spec/developer (brownfield) (optional)
├── DESIGN.md               # the design-system contract, maintained by ux-designer (optional)
├── design/                 # the PORTABLE design bundle (project-agnostic, exportable) — ux-designer
│   ├── tokens.json         #   stack-neutral tokens — the source of truth
│   ├── tokens.css          #   same tokens as CSS custom properties (buildless drop-in)
│   ├── adapters/           #   optional, generated from tokens.json for the chosen stack
│   ├── themes/             #   project theme-override files (brand instance) — written by design-binder
│   ├── preview/index.html  #   zero-build static style-guide preview (opens with no app)
│   └── BUNDLE.md           #   manifest + "adopt into a new project" recipe
├── DESIGN-BINDING.md       # CONNECTING artifact: PRD+ARCHITECTURE → the bundle (map, theme, adapter), maintained by design-binder (optional)
├── REUSE.md                # registry of reusable code, maintained by feature-developer
├── QA-PLAN.md              # durable test contract (selector map, suite scope, smoke set, NFR→check map), maintained by qa
└── QA-REPORT.md            # latest QA-gauntlet run verdict (pass/fail per check, restart history), maintained by qa
/poc/                       # disposable mock-data prototype (not under specs/; deletable)
(test code lives in the codebase per ARCHITECTURE.md §3, not under specs/)
```

## Design principles (the spine — apply to every skill, existing and new)

1. **Gating, not templates.** A skill includes only what the app needs. Patterns (multi-tenancy, queues, caching, even a feature's own file) are switched on by explicit conditions and omitted otherwise. Cheaper to add later than to rip out. When unsure, default to the simpler/off option.
2. **One concept, one owner; clean seams.** Product scope → PRD. Feature behaviour → feature specs. Technical design → architecture. The **portable** visual/interaction language → `ux-designer` (the project-agnostic bundle: `DESIGN.md` + `specs/design/`). **Binding** that portable system to *this* project (screen→component map, project theme override, stack-adapter choice) → `design-binder` (`DESIGN-BINDING.md` + `specs/design/themes/`). The seam between them is load-bearing: the bundle carries **no** project specifics (so it stays exportable), the binding carries **all** of them (so the project is fully wired) — neither writes the other's file. Disposable proof-of-concept → the PoC (`/poc/` code + `POC-NOTES.md` hints), in either mode — but the PoC only ever *reads* the real code as context and *mocks* its seam; it never builds, copies, or modifies real code (greenfield hands the foundation to the scaffolder; brownfield hands the change to `feature-spec` → `feature-developer`). Implementation + reuse registry → `feature-developer` (code/tests + `REUSE.md`). No skill redoes another's job; they **link and trace**, never duplicate. If work surfaces a decision that belongs upstream, route it back to the owning skill rather than diverging silently — the developer routes behaviour gaps to `feature-spec`, technical-design gaps to `web-app-architect`, and visual-design gaps (a missing component/token) to `ux-designer` instead of improvising. Note the deliberately fine seam between the PoC and `ux-designer`: the PoC previews *what using the product feels like* (clickable, mock data, throwaway); `ux-designer` defines *the visual language* and previews it as a *static* style guide (durable). They're complementary, not duplicative.
3. **On first creation, interview to capture intent; thereafter ask only what matters.** The two are one rule, scoped by mode. **When a skill's primary artifact does not yet exist (bootstrap), enter interview mode**: extract everything the conversation and upstream artifacts already state, then *actively interview the user* to pull out intent the artifact would otherwise have to guess — asking follow-ups until no assumption that would materially shape the artifact can still be resolved by asking. The artifact is the durable source of truth; it is worth a real conversation to get the best possible data captured *at the moment of creation*, because a thin artifact silently corrupts every stage downstream of it. The interview is still **impact-tiered** (lead with the few high-impact, artifact-shaping questions; never invent those — the hard "ask before generating" threshold still holds) and still **bounded**: it ends when remaining unknowns are genuinely lower-impact or unknowable-by-asking, which then become recorded assumptions / Open Questions the user confirms — not an infinite form. Don't re-ask what an upstream artifact already settles (Step 0 reads it first); confirm an inference rather than interrogate. **In consult & maintain mode (the artifact already exists), the posture flips back to "ask only what matters"**: treat the artifact as agreed, touch only what changed. Skills whose intent comes mostly from *upstream artifacts* rather than the user (e.g. the developer/scaffolder/qa) interview narrowly — only the genuinely user-owned unknowns — and otherwise derive and confirm.
4. **Read upstream first (Step 0).** A skill reads the previous stage's artifact before asking anything, so it never re-asks what's already settled (e.g. the architect derives project name, auth, tenancy, integrations from the PRD).
5. **Prove it.** "Done" means observable, not "files exist." Outputs carry acceptance criteria / checkable conditions; the scaffolder must boot, pass the CI gate, and meet the architecture's Phase-0 criteria.
6. **Traceability.** Stable IDs and parent/child links. PRD requirements are `FR-0X`; feature specs extend them as `FR-0X.Y`; the PRD §7 index links to each feature spec. `⚠️ TBD` inline mirrors into an Open Questions register. Every doc has a Changelog.
7. **Two modes.** Each skill checks for its output file: absent → **bootstrap**; present → **consult & maintain** (update the relevant section + Changelog, never let code and doc drift).
8. **Disposable work hands off via a cheap interface, not via its output.** Where a stage produces throwaway artifacts (the PoC's `/poc/` spike), the *next* stage consumes a small, durable hints doc (`POC-NOTES.md`) — never the throwaway code itself. This keeps the seam clean and is the token/quota-efficient choice: re-reading and salvaging a spike's source (then ripping its shortcuts back out) is the expensive path. In **brownfield** mode the same rule is load-bearing in the other direction: the PoC *reads* the real app's code to learn its integration seam but **never copies it into `/poc` or imports/edits it** — the seam is **mocked** in the spike and **named** in the notes, so the disposable code can be deleted without loss and the real wiring happens in `feature-developer`. The advisory doc never overrides an upstream owner's doc — on conflict, the authoritative spec (e.g. `ARCHITECTURE.md`) wins.

## SKILL.md authoring conventions

- **Frontmatter:** `name` (kebab-case, matches the folder) + a **deliberately pushy `description`**. Descriptions under-trigger by default, so state *what it does*, *when to use it*, concrete trigger phrases, how it disambiguates from sibling skills, and its precondition (which file must exist). Include examples — they measurably improve activation.
- **Length:** keep `SKILL.md` under ~500 lines. Move depth into a `resources/` subfolder referenced situationally ("when X, consult resources/y.md"). Only `name` + `description` load until the skill triggers.
- **Standard sections we use:** precondition / two-modes; Step 0 (read upstream); an **interview-mode** gather on first creation (impact-tiered: lead with high-impact, artifact-shaping questions, follow up until no material assumption remains askable, then default/record the rest — see Principle 3); a decision-gate table or gating rule; a verifiable output template; a scope boundary ("up/down/sideways — what this skill does NOT own"); maintenance rules; examples — including at least one example of the skill **declining** to over-do its job (e.g. not speccing a trivial feature, not bolting on multi-tenancy) and ideally one showing the interview pulling out intent the user didn't volunteer.
- **Shape:** structured input → checkable output; compose via shared files under `specs/`.

## How we develop/improve skills (the loop)

1. Read the skill-creator conventions (`/mnt/skills/examples/skill-creator/SKILL.md` in the authoring environment) before writing a `SKILL.md`.
2. Draft the skill, then **write 1–2 test prompts and dry-run them** by following the SKILL.md as if you were the agent. Pick prompts that pull behaviour in opposite directions (a gates-heavy case and a gates-light/ambiguous case).
3. Note the rough edges and **patch the SKILL.md**. This loop already surfaced and fixed: an explicit ask-vs-assume threshold, handling of "light"/borderline gates, an audit-logging gate, leading the questionnaire with high-impact gates + biasing to the simpler default, making the throwaway/durable seam explicit so the scaffolder reads only `POC-NOTES.md` (not the spike code) and the architecture wins on conflict, and — across all ten skills — promoting the first-creation gather into an explicit **interview mode** (extract, then interview until no material assumption remains askable) so the durable artifact captures the best possible intent at the moment it's created, while consult & maintain stays terse.
4. After any change to a description or gate, re-run the relevant test prompt as a regression check.

## Roadmap — planned skills

These extend the pipeline past the build loop into acceptance and release (QA is now built — see the callout below). Build each on the same spine (read the upstream artifact, gate, prove it, two modes, clean seams). Rough suggested order: UAT → release.

> **`quality-requirements` is built** (the non-functional companion to `product-requirements`, sitting after the PRD and alongside the architect). It writes `specs/QUALITY.md` — the **source of truth for non-functional targets** as stable `NFR-0X` ids, each with a *measurable* target: the quality attributes (performance, scalability, availability/SLA, security, reliability, maintainability, accessibility, observability) plus a **first-class GDPR / data-protection section** gated on whether the app handles personal data. Gating is **SaaS-aware**: a SaaS gets an always-on baseline floor (tenant-isolation, baseline availability, perf budget, WCAG AA, observability, GDPR-on) with heavier dimensions (HA/multi-region, scale/throughput, DR RPO/RTO, extra compliance regimes, real-time latency, cost budgets) gated on explicit triggers; a trivial/internal tool falls back to aggressive-off. Clean-seamed and **bidirectional with `web-app-architect`**: the architect reads `QUALITY.md` and designs to each target (recording the *mechanism* in `ARCHITECTURE.md`), and an infeasible/expensive target routes back here to renegotiate — the *target* lives in `QUALITY.md`, the *mechanism* in `ARCHITECTURE.md`, and the two must never drift on a shared number. It owns targets only — not product scope (`product-requirements`), not feature behaviour (`feature-spec`), not the technical how (`web-app-architect`), not the verification (each NFR names a `Verified by: qa` handoff, but authoring/running the check is the `qa` skill's job).
>
> **`feature-developer` is built** (the `feature-spec` → `feature-developer` build-loop pair is complete). It implements one feature from its `specs/features/<feature>.md` + `specs/ARCHITECTURE.md` as a tested vertical slice that honours the architecture's invariants/gates/principles and proves the spec's acceptance criteria, and it maintains `specs/REUSE.md` — a reuse registry it reads first (build from pre-conceived context) and updates after. Seam: builds to spec; routes behaviour gaps back to `feature-spec`, technical-design gaps to `web-app-architect`, and visual-design gaps to `ux-designer` rather than improvising.
>
> **`ux-designer` is built** (the optional design-system stage, sitting after the PoC and before the scaffolder). It produces a **portable, project-agnostic design bundle**: `specs/DESIGN.md` + `specs/design/` (stack-neutral `tokens.json`/`tokens.css` as the source of truth, optional generated stack adapters, a **zero-build** static style-guide preview that opens with no app, and `BUNDLE.md`). The bundle is self-contained and **exportable** — copy it into any project and it renders standalone — which is why it carries no project specifics (it reads the PRD only for persona/mood). It's gated (an admin panel / dev tool / off-the-shelf-kit app gets none) and clean-seamed: the scaffolder adopts the bundle, `poc-developer` can consume `tokens.css` pre-scaffold, every `feature-developer` slice conforms, and missing components/tokens route back here. Distinct from the PoC (product-feel spike, clickable, throwaway) vs. this (visual language, static preview, durable).
>
> **`design-binder` is built** (the optional *connecting* stage, sitting right after `ux-designer` and before the scaffolder). It **hard-wires the PRD + ARCHITECTURE to the portable bundle**, turning the inherited generic system into *this project's* design: it writes `specs/DESIGN-BINDING.md` (a screen/flow→component map — which bundle components/tokens/layout realize each PRD flow + architecture route — plus the chosen stack adapter the architecture consumes) and a project **theme-override** token file under `specs/design/themes/` (the brand instance, layered on the bundle without polluting it). Requires both the bundle (`specs/DESIGN.md`) and `specs/PRD.md`; reads `ARCHITECTURE.md` for the stack/routes. Gated like the design system (no bundle → nothing to bind; trivial app → skip). Clean-seamed: it never edits the bundle (that stays exportable) and never re-decides the visual language (missing component/token → route to `ux-designer`) or product scope (→ `product-requirements`); the scaffolder and every `feature-developer` slice read **both** the bundle (the language) and the binding (how this project uses it). This is the deliberate two-artifact split the suite wanted: **one exportable design artifact (`ux-designer`) inheritable into any project, and one connecting artifact (`design-binder`) that contextualizes it into the specific project — both consumed by the developer skills.**
>
> **`qa` is built** (the QA gauntlet — the verification/gate stage that closes the build loop, sitting after features are built and before release). It tests the app as a **black box** (rendered UI + published API; never imports source or touches the DB) across four **gated** passes — **e2e feature/functionality** (always on), **smoke-set flagging** (always on; `@smoke` tag in-suite + listed in the plan), a **bounded non-destructive API-vulnerability scan** (gated on an API surface or sensitive/multi-tenant data; authz/IDOR/cross-tenant, authn gaps, injection, headers, secret leakage, rate-limit presence — never DoS/destructive), and **performance testing** (gated on a perf `NFR-0X` existing in `QUALITY.md`). It turns each `QUALITY.md` `NFR-0X` `Verified by: qa` handoff into a runnable check, requires the CI gate green **first**, then runs and **restarts the whole gauntlet on any failure until green** — owning the verdict but **routing the fix** (bug → `feature-developer`; testID → `feature-developer`, scheme/test-env → `web-app-architect`; target → `quality-requirements`). The **test-centered seam** is load-bearing: the architect owns the **testID scheme** + **test-environment contract** (`ARCHITECTURE.md` §7a), `feature-developer` **emits** testIDs per the scheme (keyed to `FR-0X.Y`), and `qa` **derives** selectors from the scheme — stable, traceable, no DOM-scraping. Writes `specs/QA-PLAN.md` (durable test contract, the QA analogue of `REUSE.md`) + `specs/QA-REPORT.md` (latest verdict). Owns verification + the gate only — not targets (`quality-requirements`), behaviour (`feature-spec`), the technical how (`web-app-architect`), or fixing (`feature-developer`).

- **`uat`** — turns acceptance criteria from the PRD and feature specs into a user-runnable UAT checklist, and records sign-off before release. Reads: PRD + feature specs (their acceptance criteria). Writes: a UAT checklist + sign-off record. Seam: validates against stated criteria; doesn't invent new ones (gaps go back to the PRD/feature spec).
- **`release-manager`** — promotes through environments (local → staging → production) only when the QA gate is green; manages versioning, changelog aggregation, and the deploy flow. Reads: architecture's environments/release section + QA status. Writes: release notes / version bumps; performs the promotion. Mind the action-safety boundary — destructive or irreversible deploy steps need explicit human confirmation.
